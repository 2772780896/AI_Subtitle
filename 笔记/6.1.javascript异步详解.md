# JavaScript 异步详解

## 一、为什么需要异步？

JavaScript 是**单线程**的，同一时刻只能做一件事。

如果发一个网络请求要 3 秒，没有异步的话，整个页面就卡死 3 秒，用户什么都点不了。

异步的本质就是：**把耗时的事丢给浏览器/Node 去做，主线程继续干别的，做完了再回来处理结果。**

---

## 二、异步的三种写法（演进历史）

### 1. 回调函数（最原始）

```javascript
// 耗时操作完成后，调用你传的回调函数
setTimeout(() => {
    console.log('3秒后执行');
}, 3000);

// 典型场景：读文件
fs.readFile('data.txt', (err, data) => {
    // data 就是读到的内容
});
```

**问题**：多个异步有先后依赖时，回调层层嵌套 → **回调地狱**

```javascript
getUser(id, (user) => {
    getOrders(user, (orders) => {
        getDetails(orders, (details) => {
            // 已经嵌套三层了...
        });
    });
});
```

### 2. Promise（解决回调地狱）

Promise 是一个**凭据**，代表"一个未来才会有的结果"。

```javascript
// 创建 Promise
const p = new Promise((resolve, reject) => {
    // resolve = 成功时调用，reject = 失败时调用
    setTimeout(() => {
        resolve('数据加载完成');
        // 如果出错：reject('出错了');
    }, 3000);
});

// 使用 Promise
p.then(result => {
    console.log(result);  // '数据加载完成'
}).catch(error => {
    console.log(error);
});
```

**链式调用解决嵌套问题**：

```javascript
getUser(id)                        // 返回 Promise
    .then(user => getOrders(user)) // 返回 Promise
    .then(orders => getDetails(orders)) // 返回 Promise
    .then(details => console.log(details))
    .catch(err => console.log(err));   // 任何一步出错都能捕获
```

**Promise 的三种状态**：

```
pending（进行中）→ fulfilled（成功）→ 调用 .then
                 → rejected（失败）→ 调用 .catch
```

状态一旦改变就不可逆，且只会变一次。

### 3. async / await（最终形态，Promise 的语法糖）

```javascript
async function loadData() {
    const user = await getUser(id);        // 等 getUser 完成
    const orders = await getOrders(user);  // 再等 getOrders 完成
    const details = await getDetails(orders); // 再等
    console.log(details);
}
```

- `async` 标记函数 → 这个函数**自动返回 Promise**
- `await` 写在函数内部 → **暂停执行，等 Promise 完成再继续**
- 错误用 `try/catch` 捕获（和同步代码一样的写法）

```javascript
async function loadData() {
    try {
        const result = await someAsyncTask();
        console.log(result);
    } catch (err) {
        console.log('出错了:', err);
    }
}
```

## 三、Promise 与 async 的对应关系

async 函数就是帮你省掉 `new Promise((resolve, reject) => {...})` 这层壳：

| Promise | async 函数 |
|---|---|
| `resolve(值)` | `return 值` |
| `reject(错误)` | `throw 错误` |
| `.then(result => ...)` | `await` 拿到的值 |
| `.catch(err => ...)` | `try/catch` 捕获的 err |

```javascript
// Promise 写法
function getData() {
    return new Promise((resolve, reject) => {
        if (成功) resolve('结果');
        else reject('出错了');
    });
}

// async 写法（完全等价）
async function getData() {
    if (成功) return '结果';       // 等价于 resolve
    else throw '出错了';           // 等价于 reject
}
```

## 四、关键概念辨析

### await 不是"阻塞"

```
阻塞（同步）：我站在这等，全世界等我。主线程卡死。
await（异步等待）：我先去干别的，这个事做完了叫我。主线程不卡。
```

`await` 只是让**当前 async 函数内部**暂停，**外部调用者不受影响**。

### async 函数自动返回 Promise

```javascript
async function fn() {
    return 'hello';
}

// 等价于：
function fn() {
    return Promise.resolve('hello');
}

// 所以调用者要用 .then 或 await 来拿结果
fn().then(result => console.log(result));  // 'hello'
```

### 没有 await 会怎样？

```javascript
async function test() {
    someAsyncTask();   // 没写 await
    console.log('下一行');  // ← 立刻执行，不等 someAsyncTask 完成
}
```

不写 `await` = 不等待 = 立刻跑下一行。异步任务在后台继续跑，但顺序无法保证。

### 不需要等的异步怎么写？

直接不写 await，异步任务照样执行，只是你不等结果：

```javascript
async function run() {
    sendLog('用户登录了');   // 不 await，发出去就行
    console.log('继续处理');  // 立刻执行
}
```

多个互不依赖的任务可以同时发起，比串行更快：

```javascript
async function run() {
    const taskA = fetchA();   // 同时发起，不 await
    const taskB = fetchB();   // 同时发起

    const resultA = await taskA;  // 这时再等结果
    const resultB = await taskB;
}
```

## 五、事件循环（Event Loop）

异步代码在后台到底是怎么运转的？

```
┌─────────────────────────────────────────────┐
│              调用栈（Call Stack）              │
│         同步代码从上到下依次执行               │
│    遇到异步 → 交给宿主环境（浏览器/Node）       │
└──────────────────┬──────────────────────────┘
                   │ 异步完成后
                   ▼
┌─────────────────────────────────────────────┐
│            任务队列（Task Queue）              │
│    异步完成后的回调函数在这里排队              │
│    调用栈空了 → 从队列取一个放进调用栈执行      │
└─────────────────────────────────────────────┘
```

**执行流程**：

1. 同步代码依次入栈执行
2. 遇到 `setTimeout`、`fetch`、`fs.readFile` 等异步操作 → 交给宿主环境处理
3. 宿主环境在后台完成异步操作
4. 完成后把回调函数放入**任务队列**
5. 调用栈空了 → 从队列头部取一个回调放入调用栈执行
6. 循环往复

**所以异步不是"同时做"，而是"先挂着，做完了排队等执行"。**

## 六、错误捕获与传递

### 三种风格的错误处理

```javascript
// 回调风格
fs.readFile('file.txt', (err, data) => {
    if (err) { console.log('出错:', err); return; }
    console.log(data);
});

// Promise 风格
somePromiseTask()
    .then(data => console.log(data))
    .catch(err => console.log('出错:', err));

// async/await 风格
async function run() {
    try {
        const data = await somePromiseTask();
        console.log(data);
    } catch (err) {
        console.log('出错:', err);
    }
}
```

### throw 能丢什么？

**随便丢**，不一定要 Error 对象：

```javascript
throw '出错了';           // 可以
throw 42;                 // 可以
throw { msg: 'boom' };   // 可以
throw new Error('boom'); // 可以，推荐写法（自带 stack 堆栈信息）
```

### 错误传递：冒泡机制

错误会**沿着调用链往上冒泡**，直到有人接住它：

```javascript
async function step1() {
    throw 'boom';           // ① 这里出错
}

async function step2() {
    await step1();          // ② await 把 reject 变成 throw，继续往上抛
    console.log('不会执行');
}

async function step3() {
    try {
        await step2();      // ③ 在这里捕获
    } catch (err) {
        console.log(err);   // ④ 'boom'
    }
}
```

传递路径：`throw` → step1 的 Promise 变成 rejected → step2 的 await 收到错误 → step2 的 Promise 也变成 rejected → step3 的 try/catch 接住

**每一层都可以选择接住（catch）或者继续往上扔，像烫手山芋一样。**

### 没人接会怎样？

```javascript
async function run() {
    throw 'boom';
}
run();  // 没 await，没 .catch → 没人接 → Unhandled Promise Rejection
```

错误冒泡到最顶层没人处理，Node.js 会报警告甚至崩溃退出。**所以异步操作一定要确保有人处理错误。**

## 七、常见误区

### 误区1："异步就是不阻塞"

不完全对。异步**本身**不阻塞主线程，但 `await` 会让 async 函数内部"暂停等待"。
关键区别：这个"暂停"不影响主线程处理其他事情。

### 误区2："async 函数执行得更慢"

不是更慢，是**看起来更慢**（因为要等）。实际上它让出了主线程，
整体性能比阻塞更好。

### 误区3："不写 await 异步任务就不执行了"

会执行，只是你**不等它**。任务照样在后台跑，但你不等结果就继续往下走了。

### 误区4："Promise 让异步变同步了"

没有。Promise 只是让你用**同步的写法**写异步代码，本质还是异步的，
主线程不会被阻塞。

---

## 八、一张图总结

```
回调函数          →  能用，但嵌套多了就乱
Promise          →  链式调用，解决嵌套
async/await      →  同步写法，最直观
async ≈ Promise   →  return = resolve, throw = reject
await ≈ .then     →  拿结果的方式，二选一
事件循环          →  异步的底层运行机制（调用栈 + 任务队列）
await ≠ 阻塞     →  只是挂起当前函数，主线程照常工作
错误冒泡          →  沿调用链往上传，直到被 catch 接住
没人接错误        →  程序报警甚至崩溃
```
